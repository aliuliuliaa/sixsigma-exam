1.1 典型项目系统架构 架构

一个完整的RX3i项目通常由三层架构组成：

层级组件功能典型设备
现场层传感器/执行器采集信号、执行动作温度变送器、电机、阀门

监控层Cimplicity SCADA画面监视、报警管理、数据记录Cimplicity Server+Viewer
实际项目架构示例

以一条生产线为例，系统架构如下：

  • RX3i CPU：CPE330，2MB用户内存，背板总线控制
  • I/O模块：IC695系列数字量16点输入x4块、16点输出x3块、模拟量8通道输入x6块、4通道输出x2块
  • 通信：CPU以太网口接Cimplicity Server + 触摸屏；串口模块接变频器(Modbus RTU)
  • 上位机：1台Cimplicity Server + 2台Viewer，双网卡分别接PLC网和管理网
  • 远程I/O：通过Profinet或Genius总线扩展远程站

CPU选型对比

型号用户内存支持I/O以太网口适用场景

  • IC695PSD140：140W直流电源，适合大型机架或模拟量多的场合
  • 电源模块必须安装在机架第1槽，CPU通常在第2槽
注意电源功率不足会导致I/O模块随机掉线或通信中断。计算总功耗时需加上CPU本身的背板功耗（约5-8W），并留30%余量。
1.3 PME工程结构总览 工程

在PME中创建项目时，整个工程的结构如下：

PME工程结构
Project（工程，最多32个Target） Target 1: PAC控制器（如CPE330） Hardware Configuration（硬件组态 HWC） CPU模块（CPE330, 槽2） 电源模块（PSD140, 槽1） I/O模块（IC695系列, 槽3-N） 通信模块（如ETC001 PNIO控制器） Logic（程序逻辑） Main Block（主程序块） Program Block（子程序块） UDFB（用户自定义功能块） Variable List（变量表） Target 2: HMI（如QuickPanel+） 画面配置 变量映射

实际项目中，一个Project可以包含多个Target。例如一台PME工程里同时包含PLC的Target和触摸屏的Target，方便统一管理。

项目规范建议建议每个产线或设备段建立独立Project，不要把多条产线塞进同一个工程。Project过大（超过5MB）会导致编译和下载变慢。
1.4 网络拓扑设计 网络

典型RX3i项目网络拓扑：

网络协议连接对象速率
控制网SRTP / EGDCPU ↔ Cimplicity Server100Mbps
现场网ProfinetCPU ↔ 远程I/O站100Mbps
串行网Modbus RTUCPU ↔ 变频器/仪表9.6-115.2kbps
管理网OPC UACimplicity ↔ MES/ERP1Gbps
双网卡分离方案

CPE330有2个以太网口，推荐配置：

  • ETH1（192.168.1.x）：接控制网交换机，连接Cimplicity Server和触摸屏
  • ETH2（192.168.2.x）：接Profinet远程I/O网络
  • 两个网络物理隔离，避免广播风暴互相影响
常见问题不要将Cimplicity上位机、办公电脑和远程I/O混接在同一个交换机上。Profinet的实时报文会被普通数据流量干扰，导致I/O通信超时。
第二章

程序框架与执行流程

程序块类型与调用关系、扫描周期执行顺序、程序架构设计规范
2.1 四种程序块类型与用途 框架

RX3i程序由四种块组成，各有不同用途：

块类型功能执行方式项目中的角色
Main主程序入口每次扫描执行调用各子程序块，不写具体逻辑
Program Block子程序被Main调用时执行按功能划分：原料、加工、包装等
UDFB用户功能块被调用时执行封装标准设备控制逻辑（电机、阀门等）
Interrupt Block中断程序事件触发时执行高速计数、紧急停机、定时采样
典型Main块结构
// Main块 - 只做调用，不写逻辑 CALL PB_SYSTEM_INIT // 系统初始化（上电一次性执行） CALL PB_MOTOR_CONTROL // 电机控制 CALL PB_VALVE_CONTROL // 阀门控制 CALL PB_PID_CONTROL // PID回路 CALL PB_ALARM_PROCESS // 报警处理 CALL PB_COMM_PROCESS // 通信处理 CALL PB_HMI_INTERFACE // 上位机接口
编程规范Main块中只写CALL指令，不写具体控制逻辑。每个功能模块独立为Program Block，方便调试、修改和复用。
2.2 扫描周期与执行流程 执行

PLC每个扫描周期的执行顺序：

扫描周期执行流程
1. 读取输入（%I/%AI 刷新） 2. 执行通信服务（SRTP/EGD报文处理） 3. 执行Main块 → 按顺序调用各Program Block 4. 执行后台任务（如PID运算） 5. 检查中断条件 → 触发则执行中断块 6. 写入输出（%Q/%AQ 刷新） 7. 诊断检查 → 回到步骤1，循环执行

典型扫描周期：空载约1-3ms，中等程序约5-15ms，大型程序不超过30ms。

性能注意如果扫描周期超过50ms，需要检查：是否有大量浮点运算、是否有死循环、UDFB是否被过度调用。可以用PME在线监控CPU负载（%T指标）。

Scan Sets（扫描集）配置

当程序量很大时，可以将不同Program Block分配到不同扫描集，分时执行：

Scan Set分配的块执行频率
#1（默认）Main + 关键控制块每次扫描
#2通信处理块每2次扫描
#3数据记录块每5次扫描

2.4 系统状态机设计 状态机

项目中最核心的设计是系统运行状态机，控制整条产线的运行模式：

状态码状态名条件动作
0INIT（初始化）上电/复位变量清零、设备回原点
10IDLE（待机）初始化完成等待启动命令
20RUNNING（运行）启动命令+无故障各段正常运行
30HOLDING（保持）暂停命令输出保持当前位置
90FAULT（故障）任意故障触发按故障等级处理
99ESTOP（急停）急停按钮所有输出切断
ST状态机实现
// 系统状态机（ST编程） CASE IV_SYS_STATE OF 0: // INIT CALL_INIT_SEQUENCE(); IF INIT_DONE THEN IV_SYS_STATE := 10; END_IF; 10: // IDLE IF DI_START_CMD AND NOT ANY_FAULT THEN IV_SYS_STATE := 20; END_IF; 20: // RUNNING IF DI_PAUSE_CMD THEN IV_SYS_STATE := 30; ELSIF ANY_FAULT THEN IV_SYS_STATE := 90; END_IF; 30: // HOLDING IF DI_START_CMD THEN IV_SYS_STATE := 20; ELSIF ANY_FAULT THEN IV_SYS_STATE := 90; END_IF; 90: // FAULT IF DI_RESET_CMD AND NOT ANY_FAULT THEN IV_SYS_STATE := 10; END_IF; 99: // ESTOP IF NOT DI_ESTOP AND DI_RESET_CMD THEN IV_SYS_STATE := 90; END_IF; END_CASE;
设计要点状态机放在Main块的最前面执行，确保每次扫描都先判断当前状态，再决定是否执行各功能块的逻辑。各Program Block内部也根据IV_SYS_STATE来决定是否执行。
第三章

变量系统与内存规划

内存区域分配、变量类型与命名、间接寻址、项目内存规划实战
3.1 内存区域全览与项目分配 内存

RX3i的内存区域设计是项目规划的基础，每个区域有不同用途：

区域前缀数据类型保持性项目用途
%I离散输入BOOL非保持来自I/O模块的DI信号
%Q离散输出BOOL非保持输出到I/O模块的DO信号
%M内部线圈BOOL非保持中间运算结果、标志位
%AI模拟输入INT(16位)非保持来自I/O模块的AI信号
%AQ模拟输出INT(16位)非保持输出到I/O模块的AO信号
%R寄存器INT/WORD保持PID参数、计数器、保持数据
%W扩展寄存器DINT/REAL保持浮点数据、批量数据存储
%G全局数据各种保持多CPU共享数据
%S系统状态BOOL系统系统时钟、故障状态
%SA系统保持ABOOL保持首次扫描脉冲、故障锁存
%P程序局部各种-UDFB/PB内部局部变量
项目内存规划原则
  • %I/%Q/%AI/%AQ：只用于I/O模块映射，不要在程序中直接操作
  • %M：用于不需要断电保持的中间标志
  • %R：用于需要断电保持的整数数据（PID参数、配方等）
  • %W：用于浮点数据和大量数据存储
  • 建议使用符号变量而非直接地址，提高可读性
3.2 符号变量与数据类型 变量

实际项目中应全部使用符号变量，不要直接引用地址。PME支持在Variable List中定义符号变量并绑定到内存地址：

符号变量定义示例
// 在Variable List中定义 DI_START_BTN AT %I00001 : BOOL; // 启动按钮（接DI第1点） DI_STOP_BTN AT %I00002 : BOOL; // 停止按钮（接DI第2点） DO_MOTOR_RUN AT %Q00001 : BOOL; // 电机运行输出 AI_TEMP_PV AT %AI0001 : INT; // 温度实际值（原始值0-32000） AQ_VALVE_POS AT %AQ0001 : INT; // 阀门开度输出（0-32000） RV_TEMP_PV AT %R00001 : REAL; // 温度工程量（℃） IV_STEP_NO AT %R00010 : INT; // 步序号 RV_PID_SP AT %R00020 : REAL; // PID设定值 // 不绑定地址的内部变量 RUN_MODE : BOOL; // 运行模式标志 FAULT_CODE : INT; // 故障代码

数据类型说明

类型位数范围用途
BOOL1位0或1开关量
INT16位-32768~32767%AI/%AQ原始值、计数
DINT32位-2^31~2^31-1大数值计数
REAL32位浮点数工程量、PID运算
WORD16位0~65535位操作、状态字
DWORD32位0~4294967295通信数据
3.3 模拟量工程量转换 转换

项目中模拟量处理是最常见的任务。%AI读到的是原始值（0-32000），需要转换为工程量：

工程量转换公式（ST）
// 4-20mA对应0-32000原始值，工程量0-100℃ // 原始值 6400 = 4mA = 0℃ // 原始值 32000 = 20mA = 100℃ RV_TEMP_PV := (INT_TO_REAL(AI_TEMP_PV) - 6400.0) * 100.0 / (32000.0 - 6400.0); // 通用转换函数（做成UDFB） // 输入：Raw(原始值), Raw_Min, Raw_Max, Eng_Min, Eng_Max // 输出：Eng(工程量) Eng := (INT_TO_REAL(Raw) - REAL_MIN) * (Eng_Max - Eng_Min) / (Raw_Max - Raw_Min) + Eng_Min;
注意4-20mA信号的原始值不是0开始，而是6400（4mA对应32000的20%）。0-20mA信号才是0-32000。务必确认传感器输出类型！
实际项目处理

建议将模拟量转换封装为UDFB UF_ANALOG_SCALE，每个AI点调用一次。同时加入断线检测：当原始值<3200（<2mA）时判定为传感器断线，输出故障标志。

3.4 间接寻址与数组应用 进阶

项目中经常需要处理一组同类设备（如8个电机的启停控制），使用间接寻址可以大幅简化程序：

间接寻址示例（LD中使用COMM_REQ或ST）
// ST中使用间接寻址处理8个电机 FOR IV_INDEX := 1 TO 8 DO // 读取每个电机的启动命令（%R1~%R8） START_CMD := %R[IV_INDEX]; // 调用电机UDFB（参数通过%R数组传递） UF_MOTOR(IN := START_CMD, OUT => %R[IV_INDEX + 100]); END_FOR; // LD中使用指针寄存器 // %R1000 = 基地址 // 读取 %R[%R1000] 的值
应用场景间接寻址适合：配方管理（一组参数批量读写）、多台同类设备循环处理、通信数据打包发送。但要注意边界检查，防止指针越界。
第四章

UDFB面向对象编程实战

UDFB概念与Class类比、参数传递机制、电机控制UDFB完整实例、UDFB库管理
4.1 UDFB与面向对象编程 核心

UDFB（User-Defined Function Block）是RX3i编程中最核心的概念，相当于面向对象编程中的Class（类）：

面向对象概念UDFB对应说明
Class（类定义）UDFB定义定义功能块的逻辑和变量
Object（对象/实例）UDFB实例在Program Block中调用一次产生一个实例
成员变量UDFB内部变量每个实例独立保持自己的状态
方法/成员函数UDFB内部逻辑实例被调用时执行的代码
构造函数参数输入参数调用时传入的配置参数
返回值输出参数实例执行后输出的结果
类比理解

定义一个电机控制UDFB UF_MOTOR，相当于定义了一个"电机控制类"。在程序中调用3次，就创建了3个独立的电机控制实例（电机1、电机2、电机3），每个实例有自己的运行状态、故障状态、运行时间，互不干扰。

UDFB的优势
  • 代码复用：一套逻辑给N台同类设备用
  • 易于维护：修改逻辑只需改UDFB定义，所有实例自动更新
  • 状态独立：每个实例有自己的内部变量，互不干扰
  • 封装性：外部只需知道输入输出，内部逻辑不可见
4.2 UDFB三种参数传递机制 参数

UDFB支持三种参数传递方式，选择正确的方式对程序运行至关重要：

类型关键字说明适用场景
输入参数IN调用时传入值，UDFB内可读不可写启动/停止命令、设定值
输出参数OUTUDFB内可写，调用方读取运行状态、故障标志
输入输出参数IN_OUT双向传递，UDFB内可读可写计时器累计值、步进状态
UDFB参数定义示例
// UF_MOTOR 定义 FUNCTION_BLOCK UF_MOTOR // 输入参数 IN START_CMD : BOOL; // 启动命令 IN STOP_CMD : BOOL; // 停止命令 IN FAULT_RESET : BOOL; // 故障复位 IN AUTO_MODE : BOOL; // 自动模式 // 输出参数 OUT RUN_STATUS : BOOL; // 运行状态 OUT FAULT_FLAG : BOOL; // 故障标志 OUT READY : BOOL; // 就绪状态 // 输入输出参数（有状态记忆） IN_OUT RUN_TIME : DINT; // 累计运行时间(s) IN_OUT STEP : INT; // 当前步序 // 内部变量（实例私有） VAR TMR_START : TMR; // 启动延时 INTERNAL_OK : BOOL; // 内部就绪 END_VAR
关键区别IN_OUT参数会保持状态（断电保持），即使UDFB不被调用，参数值也不会丢失。如果需要记忆上一次的运行状态（如步序、累计值），必须用IN_OUT，不能用IN+OUT替代。
4.3 电机控制UDFB完整实例 实战

以下是一个项目中最常用的电机控制UDFB完整实现，包含启动/停止/故障/运行时间累计：

UF_MOTOR 完整实现（LD+ST混合）
// === UDFB: UF_MOTOR === // 功能：标准电机控制块 // 输入：START_CMD, STOP_CMD, FAULT_RESET, AUTO_MODE // OVERLOAD_SIG(过载信号), INTERLOCK(联锁条件) // 输出：RUN_STATUS, FAULT_FLAG, READY // 输入输出：RUN_TIME(累计运行时间s) // 1. 故障处理逻辑 FAULT_FLAG := (OVERLOAD_SIG AND RUN_STATUS) OR (FAULT_FLAG AND NOT FAULT_RESET); // 2. 就绪状态 READY := NOT FAULT_FLAG AND INTERLOCK; // 3. 启动停止逻辑（自保回路） IF START_CMD AND READY AND NOT FAULT_FLAG THEN RUN_STATUS := TRUE; END_IF; IF STOP_CMD OR FAULT_FLAG OR NOT INTERLOCK THEN RUN_STATUS := FALSE; END_IF; // 4. 运行时间累计（每秒+1） IF RUN_STATUS THEN // 使用1秒系统时钟脉冲%S0002 IF %S0002 THEN // 1Hz时钟脉冲 RUN_TIME := RUN_TIME + 1; END_IF; END_IF; // 5. 输出到物理DO（在Program Block中映射） // DO_MOTOR_RUN := Motor_Instance.RUN_STATUS;

在Program Block中调用UDFB

调用实例（LD块中）
// 电机1实例 Motor_1: UF_MOTOR( START_CMD := DI_M1_START, STOP_CMD := DI_M1_STOP, FAULT_RESET := DI_M1_RST, AUTO_MODE := AUTO_MODE, OVERLOAD_SIG := DI_M1_OL, INTERLOCK := M1_INTERLOCK_OK, RUN_STATUS => DO_M1_RUN, FAULT_FLAG => M1_FAULT, READY => M1_READY, RUN_TIME := M1_RUN_TIME, STEP := M1_STEP ); // 电机2实例（同一UDFB，不同参数） Motor_2: UF_MOTOR( START_CMD := DI_M2_START, ... );
实际项目应用
  • 建立标准UDFB库：电机、阀门、PID、模拟量转换、通信处理等
  • 每个UDFB单独一个文件，命名规范统一（UF_前缀）
  • UDFB库可以在不同项目间复用，大幅提高开发效率
  • 修改UDFB定义后，所有调用实例自动更新

6.1 SRTP通信与Cimplicity连接 通信

SRTP（Service Request Transport Protocol）是GE/Emerson PLC的标准以太网通信协议，Cimplicity默认使用SRTP连接PLC：

配置步骤

  1. CPU以太网口配置：在HWC中设置IP地址（如192.168.1.10）、子网掩码（255.255.255.0）
  2. Cimplicity设备配置：在Cimplicity中添加PLC设备，选择GE SRTP驱动，填入CPU的IP地址
  3. 点位映射：在Cimplicity中创建点位，地址对应PLC的%I/%Q/%R/%AI/%AQ等
Cimplicity点位地址对应关系
// Cimplicity Point Address 格式 DI_START -> %I00001 (对应PLC %I00001) DO_MOTOR -> %Q00001 (对应PLC %Q00001) RV_TEMP_PV -> %R00001 (对应PLC %R00001, REAL占2个字) AI_TEMP -> %AI0001 (对应PLC %AI0001) AQ_VALVE -> %AQ0001 (对应PLC %AQ0001) // 在Cimplicity中配置： // Device: PLC_RX3i (SRTP, 192.168.1.10) // Point: DI_START, Address: %I00001, Type: BOOLEAN
注意事项
  • SRTP通信默认使用TCP端口18246
  • Cimplicity轮询PLC的周期建议设为500ms-1s，过快会增加CPU负载
  • 大量点位（>1000点）时建议使用EGD代替SRTP轮询
6.2 EGD生产者/消费者通信 EGD

EGD（Ethernet Global Data）是基于UDP的广播/组播通信，适合大量数据快速交换，延迟低于SRTP轮询：

EGD工作原理

角色功能数量
Producer（生产者）周期性发送数据包1个
Consumer（消费者）接收数据包多个
EGD配置实例

场景：1台PLC向3台Cimplicity Server发送1000个数据点

  1. PLC端（Producer）：在HWC中添加EGD配置，创建一个EGD页面（Page），将1000个点的数据映射到Page的预定义偏移地址
  2. 发送周期：设为100ms（10次/秒）
  3. Cimplicity端（Consumer）：每台Server配置为Consumer，接收同一个Page
  4. 优势：PLC只发1次数据包，3台Server同时接收，CPU负载极低
EGD数据映射
// PLC端EGD Page配置 // Page ID: 1 // Producer IP: 192.168.1.10 (PLC) // Consumer IP: 192.168.1.20, .30, .40 (3台Server) // 发送周期: 100ms // 数据映射到Page的偏移地址 // Offset 0: %R0001-%R0010 (10个INT, 20字节) // Offset 20: %R0101-%R0120 (20个REAL, 80字节) // Offset 100: %I0001-%I0016 (16个BOOL, 2字节) // Cimplicity端读取 // Point: EGD_TEMP, Address: EGD:Page1:Offset20, Type: REAL
6.3 Modbus TCP通信配置 Modbus

项目经常需要与第三方设备（变频器、仪表、MES系统）通信，Modbus TCP是最通用的选择：

RX3i作为Modbus TCP Master

  1. 在HWC中添加Modbus TCP Client模块（或使用CPU自带以太网口）
  2. 使用COMM_REQ指令建立连接并发送读写请求
  3. 配置从站IP、端口号（默认502）、Unit ID
Modbus TCP读取变频器状态（ST）
// 通过COMM_REQ发送Modbus请求 // 读取变频器寄存器40001-40010（运行频率、电流、电压等） // 1. 建立连接 COMM_REQ( SYSID := 1, // CPU系统ID TASK := 1, // Modbus通道号 FUNC := 16#03, // 功能码03-读保持寄存器 SLAVE := 1, // 变频器站号 START := 40001, // 起始寄存器 COUNT := 10, // 读取10个寄存器 DATA := %R500 // 数据存储到%R500-%R509 ); // 2. 数据解析 RV_FREQ := INT_TO_REAL(%R500) / 10.0; // 频率(Hz) RV_CURRENT := INT_TO_REAL(%R501) / 100.0; // 电流(A) RV_VOLTAGE := INT_TO_REAL(%R502) / 10.0; // 电压(V)

RX3i作为Modbus TCP Server

当MES或其他系统需要读取PLC数据时，将PLC配置为Server模式。第三方系统通过Modbus功能码03/06/16读写PLC的%R区域。

Modbus通信调试使用ModScan或Modbus Poll软件模拟从站/主站，先验证通信链路正常后再接入实际设备。常见问题：字节序（Big-Endian vs Little-Endian）不匹配、寄存器地址偏移1（PLC侧从1开始，Modbus协议从0开始）。
6.4 Profinet远程I/O配置 PNIO

当I/O点数超出本地机架容量，或设备分散在不同位置时，使用Profinet扩展远程I/O：

配置流程

  1. CPU侧：CPE330/CPE400内置Profinet控制器（Controller），ETH2口默认为PNIO口
  2. 远程站：安装Profinet I/O设备（如Phoenix Contact AXL系列、ETC001等）
  3. GSDML文件：将远程I/O设备的GSDML文件导入PME的HWC
  4. 组态：在HWC中添加Profinet设备，配置模块和地址映射
  5. 设备名称：每个PNIO设备必须分配唯一的Device Name
常见问题
  • Profinet设备名称和IP必须在HWC中与实际设备一致，否则无法建立连接
  • 首次调试时需要用PME的"Profinet Controller Diagnostics"检查设备是否在线
  • Profinet I/O更新周期默认1ms-4ms，比SRTP快很多
  • 不要在Profinet网络上接非PNIO设备（如办公电脑）

7.2 自动/手动无冲击切换 实战

项目中最常见的需求：操作员需要在自动和手动之间切换，切换时不能产生输出突变（无冲击切换）：

无冲击切换逻辑
// 手动/自动切换逻辑 IF DI_PID_AUTO_BTN THEN PID_AUTO_MODE := TRUE; ELSIF DI_PID_MAN_BTN THEN // 切换到手动前，将当前输出值写入手动输出 RV_MANUAL_OUT := RV_PID_CV_CURRENT; PID_AUTO_MODE := FALSE; END_IF; // 手动模式下，操作员通过上位机调整RV_MANUAL_OUT // PID块在手动模式下输出=MANUAL_CV，内部积分器跟踪 // 切回自动时，输出从当前值开始，无突变 // PID调用 PID_ISA( ... AUTO_MAN := PID_AUTO_MODE, MANUAL_CV:= RV_MANUAL_OUT );
关键点PID_ISA块在手动模式下会自动进行积分跟踪（Tracking），确保内部积分器跟踪当前手动输出值。这是内置功能，不需要额外编程。
7.3 PID参数整定方法 整定
方法步骤适用场景
经验试凑法先P后I再D，逐步加大快速调试
临界比例法纯P控制找到临界振荡，计算参数温度、压力回路
阶跃响应法给阶跃输入，记录响应曲线慢响应过程
经验试凑法实操
  1. 置I=0, D=0，逐步增大Kp直到系统出现等幅振荡，然后减小Kp到振荡消失的60-70%
  2. 逐步减小Ti（从大往小调），直到出现振荡，然后增大Ti到振荡消失的1.2倍
  3. 逐步增大Td（从小往大调），直到响应变快但不振荡。温度控制一般Td=Ti的1/4到1/8
  4. 典型参数参考：流量控制 Kp=0.5-1.5, Ti=0.1-1s；液位控制 Kp=1-5, Ti=1-10s；温度控制 Kp=2-10, Ti=5-60s

  3. 创建Interrupt Block并关联到对应的中断号
中断块配置示例
// 中断块 #1: 每10ms执行一次 // 用途：高速数据采集和滤波 // 模拟量滤波（移动平均） IV_FILTER_INDEX := (IV_FILTER_INDEX + 1) MOD 10; RV_FILTER_BUF[IV_FILTER_INDEX] := RV_TEMP_PV_RAW; // 计算平均值 RV_TEMP_SUM := 0.0; FOR IV_J := 0 TO 9 DO RV_TEMP_SUM := RV_TEMP_SUM + RV_FILTER_BUF[IV_J]; END_FOR; RV_TEMP_PV := RV_TEMP_SUM / 10.0; // 中断块执行时间必须远小于触发周期 // 10ms中断块，执行时间应<2ms
中断使用注意
  • 中断块中不要调用通信指令（COMM_REQ/SVC_REQ），可能造成死锁
  • 中断块执行时间必须远小于触发周期，否则会导致扫描周期溢出
  • 中断块中的变量修改可能与主程序产生竞争条件，需要使用中间变量缓冲

用户故障定义表
// 项目自定义故障码定义 // 1001: 温度高高报警 // 1002: 温度低低报警 // 1003: 液位高高报警 // 1004: 液位低低报警 // 1005: 电机过载 // 1006: 阀门超时 // 1007: 通信超时 // 1008: PID偏差过大 // 写入故障示例 IF RV_TEMP_PV > RV_TEMP_HH THEN SVC_REQ(21, FAULT_CODE := 1001, MSG := '温度高高', ACTION := 2); // ACTION: 1=仅记录, 2=停止PLC END_IF; // 故障复位 IF DI_FAULT_RESET THEN // 清除所有用户故障 SVC_REQ(21, FAULT_CODE := 0, ACTION := 0); END_IF;

建议在项目开始前用Excel做好地址规划，避免冲突：

地址范围用途大小

规划原则
  • 按功能分区，留足扩展空间
  • 每个PID回路预留40个%R字
  • 配方区按最大配方数x每个配方字数规划
  • 浮点数据用%W区（DINT/REAL原生32位存储）
